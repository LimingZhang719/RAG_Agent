# 知识问答会话设置功能研发计划与技术说明

## 1. 背景与目标

当前知识问答页面已经支持：

- 选择知识库后发起问答。
- 选择历史会话、新建会话。
- 流式生成回答。
- 按本次检索结果展示引用列表。
- 请求级参数中已有 `top_k` 与 `rerank_enabled` 字段，但前端没有暴露设置入口。

本次需求是在知识问答界面增加“会话设置”功能，允许用户对当前会话执行以下操作：

- 删除当前会话。
- 设置当前会话的 `top_k`。
- 设置当前会话是否启用 rerank。
- 设置当前会话的 `temperature` 值。
- 设置当前会话的系统提示词。

目标是把这些设置作为“会话级配置”持久化。用户切换会话后，应自动恢复该会话自己的设置；继续提问时，后端应使用该会话的配置参与检索和生成。

## 3. 设计原则

1. 会话设置必须持久化，刷新页面后不丢失。
2. 设置只作用于当前会话，不影响其他会话，也不改全局默认配置。
3. 尽量复用 `chat_sessions.metadata`，避免为少量灵活配置新增表。
4. 删除会话应采用硬删除还是软删除需要明确。本方案建议前端操作为“删除”，后端采用硬删除；如果后续需要回收站，再改为 `is_archived` 软删除。
5. 参数校验放在后端 schema 层，前端只做交互级限制。
6. 系统提示词要限制长度，避免过长输入影响上下文窗口、成本和响应稳定性。

## 5. 会话设置数据结构

建议复用 `chat_sessions.metadata` 存储如下结构：

```json
{
  "settings": {
    "top_k": 8,
    "rerank_enabled": false,
    "temperature": 0.7,
    "system_prompt": "你是企业知识库问答助手..."
  }
}
```

建议默认值：

| 字段 | 类型 | 默认值 | 校验范围 | 说明 |
| --- | --- | --- | --- | --- |
| `top_k` | integer | `settings.rag_top_k` | 1-50 | 返回给生成模型的最终检索切片数 |
| `rerank_enabled` | boolean | `settings.rerank_enabled` | true/false | 是否对召回结果进行重排 |
| `temperature` | number | 0.7 | 0-2 | 模型生成随机性 |
| `system_prompt` | string | 当前默认模板 system | 0-4000 字符 | 为空时使用默认系统提示词 |

说明：

- `system_prompt` 为空字符串时，不建议直接传空系统提示词，而是回退到默认模板 system。
- `top_k` 对最终返回切片数生效；keyword recall 的 `keyword_top_k` 仍使用全局配置，除非后续单独开放。
- `rerank_enabled` 为 `null` 时才使用全局默认；保存到会话设置后应是明确 boolean。

## 6. 后端技术方案

### 6.1 Schema 设计

在 `backend/app/schemas/chat.py` 增加：

```python
class ChatSessionSettings(BaseModel):
    top_k: int = Field(default=settings.rag_top_k, ge=1, le=50)
    rerank_enabled: bool = Field(default=settings.rerank_enabled)
    temperature: float = Field(default=0.7, ge=0, le=2)
    system_prompt: str | None = Field(default=None, max_length=4000)


class ChatSessionUpdate(BaseModel):
    title: str | None = Field(default=None, max_length=255)
    kb_ids: list[UUID] | None = None
    settings: ChatSessionSettings | None = None
```

`ChatSessionResponse` 建议增加：

```python
settings: ChatSessionSettings
```

为了兼容已有数据，读取时从 `metadata_["settings"]` 合成响应；没有配置时返回默认设置。

### 6.2 Service 设计

在 `chat_service.py` 增加：

- `get_session_settings(chat_session) -> ChatSessionSettings`
- `update_session(session, session_id, user, payload) -> ChatSession`
- `delete_session(session, session_id, user) -> None`
- `merge_session_metadata_settings(metadata, settings) -> dict`

删除逻辑建议：

- 确认会话归属当前用户。
- 删除 `ChatSession`。
- 依赖 ORM relationship 的 `cascade="all, delete-orphan"` 删除消息。
- 同时删除关联 `RetrievalLog`，因为当前 `RetrievalLog.session_id` 不是 ORM cascade 关系的一部分。若数据库外键没有级联，必须显式删除。

### 6.3 API 设计

新增或扩展路由：

```http
GET /chat/sessions/{session_id}
PATCH /chat/sessions/{session_id}
DELETE /chat/sessions/{session_id}
```

建议响应：

- `GET` 返回 `ChatSessionResponse`
- `PATCH` 返回更新后的 `ChatSessionResponse`
- `DELETE` 返回 `204 No Content`

错误处理：

- 不存在：404
- 非本人会话：403
- 参数非法：422
- 删除当前正在流式生成的会话：前端禁用删除按钮；后端无需特殊锁，保留普通一致性即可。

### 6.4 RAG 服务改造

`rag_service.stream_chat` 当前接收请求级 `top_k` 和 `rerank_enabled`。改造后推荐优先级：

1. 请求显式传入的参数。
2. 当前会话持久化设置。
3. 全局默认设置。

伪代码：

```python
session_settings = get_session_settings(chat_session)
effective_top_k = top_k or session_settings.top_k or settings.rag_top_k
effective_rerank = (
    rerank_enabled
    if rerank_enabled is not None
    else session_settings.rerank_enabled
)
effective_temperature = session_settings.temperature
effective_system_prompt = session_settings.system_prompt
```

注意：前端保存会话设置后，发送消息时可以不再重复传 `top_k/rerank_enabled/temperature/system_prompt`，后端直接从会话读取。为兼容未创建会话的首问，可以在首问 payload 中带当前设置，后端创建会话时把设置写入 metadata。

因此 `ChatStreamRequest` 建议增加：

```python
temperature: float | None = Field(default=None, ge=0, le=2)
system_prompt: str | None = Field(default=None, max_length=4000)
```

### 6.5 Prompt 改造

当前 `build_messages(question, context)` 固定使用模板 system。

建议改为：

```python
def build_messages(
    question: str,
    context: str,
    system_prompt: str | None = None,
) -> list[dict[str, str]]:
```

规则：

- `system_prompt` 有有效内容时，使用自定义 system。
- 否则使用默认模板 system。
- user prompt 仍沿用当前模板，保证上下文和问题格式稳定。

### 6.6 LLM Client 改造

当前 `LLMClient.stream_chat(messages)` 不接受模型参数。

建议改为：

```python
async def stream_chat(
    self,
    messages: list[dict[str, str]],
    temperature: float | None = None,
) -> AsyncGenerator[str, None]:
```

OpenAI-compatible 调用：

```python
stream = await self._client.chat.completions.create(
    model=self._model,
    messages=messages,
    temperature=temperature,
    stream=True,
)
```

注意：

- 如果部分兼容接口不支持 `temperature=None`，应只在非空时传入。
- `FakeLLMClient` 需要同步签名。

### 6.7 数据库迁移

如果复用现有 `chat_sessions.metadata`，不需要新增字段迁移。

但如果决定为查询和约束更清晰而新增字段，可增加：

- `top_k`
- `rerank_enabled`
- `temperature`
- `system_prompt`

本方案建议不新增字段，理由：

- 会话设置属于低频读取的附属配置。
- 当前模型已存在 JSONB metadata。
- 后续增加更多会话设置时更灵活。

如采用 JSONB，不需要 Alembic 迁移。

## 7. 前端技术方案

### 7.1 页面入口

在知识问答顶部工具栏增加“设置”按钮，位置建议：

```text
知识库选择 | 会话选择 | 新建会话 | 设置
```

交互规则：

- 未选择会话且没有消息时，设置按钮可打开“新会话默认设置”。
- 已有当前会话时，设置按钮打开当前会话设置。
- 流式生成中禁用保存、删除。

### 7.2 设置界面形式

建议使用 Ant Design `Drawer`，原因：

- 设置项较多，比 Modal 更适合表单。
- 不打断主聊天上下文。
- 右侧抽屉符合工具设置类交互。

设置项：

- `InputNumber`：Top K
- `Switch`：是否启用 rerank
- `Slider + InputNumber`：Temperature
- `Input.TextArea`：系统提示词
- `Button danger`：删除当前会话
- `Button`：恢复默认
- `Button type="primary"`：保存

建议布局：

```text
Drawer: 会话设置

检索设置
- Top K
- 启用 Rerank

生成设置
- Temperature
- 系统提示词

危险操作
- 删除当前会话
```

### 7.3 前端状态设计

新增类型：

```ts
export interface ChatSessionSettings {
  top_k: number;
  rerank_enabled: boolean;
  temperature: number;
  system_prompt?: string | null;
}
```

`ChatSessionItem` 增加：

```ts
settings: ChatSessionSettings;
```

`ChatPage` 新增状态：

- `sessionSettings`
- `isSettingsOpen`
- `isSavingSettings`
- `isDeletingSession`

切换会话时：

1. 设置 `activeSessionId`
2. 设置 `selectedKbIds`
3. 设置 `sessionSettings`
4. 加载消息

新建会话时：

1. `activeSessionId = null`
2. 清空消息和引用
3. `sessionSettings = defaultSettings`

发送首问时：

- payload 携带当前设置，后端创建会话时保存。

发送已有会话问题时：

- 可以只传 `session_id/question/kb_ids`，后端从会话读取设置。
- 为了减少前后端状态不同步风险，建议仍传当前设置，并在后端用请求参数覆盖本次调用；保存按钮负责持久化。

### 7.4 API 封装

在 `frontend/src/api/chat.ts` 增加：

```ts
export async function updateChatSession(
  sessionId: string,
  payload: ChatSessionUpdatePayload
): Promise<ChatSessionItem>

export async function deleteChatSession(sessionId: string): Promise<void>
```

`ChatStreamPayload` 增加：

```ts
top_k?: number;
rerank_enabled?: boolean;
temperature?: number;
system_prompt?: string | null;
```

### 7.5 删除会话交互

删除当前会话流程：

1. 用户点击“删除当前会话”。
2. `Popconfirm` 二次确认。
3. 调用 `DELETE /chat/sessions/{session_id}`。
4. 从 `sessions` 状态移除该会话。
5. 如果还有其他会话，自动切到最新一条并加载消息。
6. 如果没有会话，进入新会话空状态，并恢复默认设置。

限制：

- `activeSessionId` 为空时不显示或禁用删除按钮。
- `isStreaming` 时禁用删除。

## 8. 端到端行为说明

### 8.1 新会话首问

1. 用户点击“新建会话”。
2. 用户打开设置，修改 `top_k/rerank/temperature/system_prompt`。
3. 用户发送第一个问题。
4. 后端创建会话，并将当前设置写入 `chat_sessions.metadata.settings`。
5. 后端按该设置检索和生成。
6. 前端刷新会话列表，新会话显示首问标题。

### 8.2 已有会话继续提问

1. 用户选择历史会话。
2. 前端加载该会话设置。
3. 用户发送问题。
4. 后端优先使用会话设置。
5. 引用列表仍由本次检索命中的 chunks 生成。

### 8.3 修改已有会话设置

1. 用户打开设置抽屉。
2. 修改参数并保存。
3. 前端调用 `PATCH /chat/sessions/{session_id}`。
4. 后端更新 `metadata.settings`。
5. 后续问题使用新设置；历史消息不变。

### 8.4 删除当前会话

1. 用户点击删除并确认。
2. 后端删除会话、消息、检索日志。
3. 前端从列表移除。
4. 自动进入下一会话或空会话状态。

## 11. 风险与注意事项

### 11.1 系统提示词安全风险

用户可自定义系统提示词后，可能降低“必须基于知识库回答”的约束。建议在实现中保留一段不可覆盖的安全约束，做法有两种：

1. 把用户自定义系统提示词追加到默认系统提示词后面。
2. 使用两段 system message，第一段为平台强约束，第二段为用户自定义风格要求。

推荐方案：保留平台强约束，再追加用户自定义提示词。

### 11.2 temperature 兼容性

部分 OpenAI-compatible 服务可能不接受 `temperature=None` 或某些范围。实现时只在用户设置有效值时传入，并保持默认值为 0.7。

### 11.3 删除会话的数据一致性

需要检查 `RetrievalLog` 是否存在外键级联。若没有，必须先删除 retrieval logs，再删除 session，否则可能出现外键约束错误。

### 11.4 JSONB metadata 类型兼容

历史会话的 `metadata` 可能为 `null` 或包含其他业务字段。更新 settings 时必须 merge，不应覆盖整个 metadata。

### 11.5 首问之前没有 session_id

用户在空会话状态修改设置后发送首问，此时后端还没有会话。前端必须把设置随 `/chat/stream` payload 一起发送，后端创建会话时保存。

## 12. 建议验收标准

- 用户能在知识问答页打开会话设置。
- 当前会话能保存并恢复 `top_k/rerank/temperature/system_prompt`。
- 新建会话首问后，会话设置能被持久化。
- 切换不同会话时，各自设置互不影响。
- 删除当前会话后，消息列表、引用列表、会话选择框状态正确。
- `top_k` 能影响引用列表数量上限。
- 自定义系统提示词和 temperature 能传递到模型调用。
- 未配置 settings 的历史会话仍可正常问答。
