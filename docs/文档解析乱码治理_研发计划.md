# 文档解析乱码治理研发计划

> 目标：系统性解决知识库文档切块内容中的异常空格、控制字符、乱码和抽取噪声问题，提升入库文本质量、检索准确率和问答可信度。

---

## 1. 问题描述

当前知识库中文档入库后，在切片详情中可见文本异常，例如：

```text
zlf.adm in-2107
```

具体表现包括：

- 英文单词或账号被异常插入空格，例如 `admin` 被抽取为 `adm in`。
- 文本中出现不可见控制字符，例如 ``，对应 ASCII 控制字符 `\x01`。
- PDF、Office、扫描件、表格类文档更容易出现抽取文本不连续、符号异常、段落错乱等问题。
- 这些异常内容会被直接写入 `document_blocks.content` 和 `chunks.content`，进而影响向量化、关键词检索、引用展示和最终回答。

---

## 2. 影响范围

### 2.1 对用户界面的影响

- 切片详情显示异常字符，降低用户对入库质量的信任。
- 引用片段可能包含乱码或异常断词。

### 2.2 对检索效果的影响

- 异常空格会破坏关键词匹配，例如查询 `admin` 时，数据库中存储为 `adm in`，可能导致关键词召回失败。
- 控制字符会干扰全文检索、Embedding 表达和 Rerank。
- 噪声文本会占用 chunk token，降低有效上下文比例。

### 2.3 对问答质量的影响

- LLM 接收到的上下文不干净，可能误读账号、编号、制度条款或关键实体。
- 强规则原文返回时会把脏文本直接返回给用户。

---

## 3. 当前技术链路

当前入库逻辑位于：

- `backend/app/rag/ingest.py`

核心流程：

1. 从 MinIO 下载原始文件到临时目录。
2. 使用 LlamaIndex `SimpleDirectoryReader` 自动解析文件：

```python
reader = SimpleDirectoryReader(input_files=[local_path])
documents = reader.load_data()
```

3. 使用 `SentenceSplitter` 或 `TokenTextSplitter` 切块。
4. 使用：

```python
node.get_content(metadata_mode="none")
```

得到切片文本。

5. 原样写入：

- `document_blocks.content`
- `chunks.content`

当前缺口：

- 没有对解析器输出文本做规范化。
- 没有过滤不可见控制字符。
- 没有针对 PDF 常见断词、异常空格做修复。
- 没有记录原始文本与清洗后文本的质量差异。
- 没有按文件类型选择更合适的解析器。

---

## 4. 原因分析

### 4.1 PDF 文本层天然不稳定

很多 PDF 并不保存“完整段落文本”，而是保存字符、字形和坐标。解析器需要根据字符坐标推断阅读顺序和空格位置。

因此会出现：

- 一个单词被拆成多个字符片段。
- 字符间距被误判为空格。
- 页眉、页脚、脚注、表格内容混入正文。
- 特殊字体或编码映射失败。

示例：

```text
admin -> adm in
```

这通常不是切块器主动拆词，而是解析器在抽取阶段已经输出了带空格的文本。

### 4.2 控制字符未清理

`` 属于不可见控制字符，通常来自：

- PDF 内部文本流。
- Office 文档隐藏控制符。
- 表单字段分隔符。
- 复制保护或特殊字体编码。
- 解析库保留下来的内部标记。

当前系统没有过滤 `\x00-\x1f` 等控制字符，因此这些字符会原样进入数据库。

### 4.3 通用解析器无法覆盖所有文档类型

`SimpleDirectoryReader` 的优点是简单通用，但缺点是：

- 对不同文件类型的解析质量不可控。
- 对 PDF、表格、扫描件、图片型 PDF 缺少针对性策略。
- 不方便对每类文件设置专门参数。

### 4.4 缺少入库质量治理层

当前链路是：

```text
解析 -> 切块 -> Embedding -> 入库
```

建议改造为：

```text
解析 -> 文本规范化 -> 质量检测 -> 切块 -> Embedding -> 入库
```

---

## 5. 治理目标

### 5.1 短期目标

- 清除控制字符和不可见异常字符。
- 合并明显异常空白。
- 修复常见英文单词、账号、邮箱、编号中的异常空格。
- 确保新入库文档的切片内容不再出现明显乱码。

### 5.2 中期目标

- 按文件类型选择解析器。
- 为 PDF 建立更稳定的解析策略。
- 对扫描件或图片型 PDF 识别并提示 OCR 需求。
- 增加入库质量指标和日志。

### 5.3 长期目标

- 建立可扩展的文档解析管线。
- 支持解析结果预览、人工确认和重新入库。
- 对不同业务文档模板做专门解析优化。

---

## 6. 解决方案

### 6.1 文本规范化函数

新增统一清洗函数，例如：

- `backend/app/rag/text_cleaning.py`

核心能力：

- 删除不可见控制字符。
- 保留必要换行和制表符。
- 统一全角/半角字符。
- 规范化 Unicode。
- 合并连续空格。
- 修复字母、数字、邮箱、账号、编号中的异常空格。

建议函数：

```python
def normalize_extracted_text(text: str) -> str:
    ...
```

清洗规则建议：

- 删除 `\x00-\x08`、`\x0b-\x0c`、`\x0e-\x1f`、`\x7f`。
- 保留 `\n`、`\t`，但做适度归一化。
- 将连续空格合并为单个空格。
- 修复英文字母中间异常空格：

```text
adm in -> admin
```

- 修复账号、邮箱、编号中的异常空格：

```text
zlf.adm in-2107 -> zlf.admin-2107
user @ example.com -> user@example.com
A B C-123 -> ABC-123
```

注意：不能无脑删除所有空格，否则会破坏正常英文句子。因此修复规则要限定在高置信模式内。

### 6.2 入库链路接入

在 `ingest.py` 中切块前或切块后接入清洗。

推荐策略：

1. 解析器输出后，先对每个 `LlamaDocument.text` 做基础清洗。
2. 切块后，对每个 chunk 内容再做一次轻量清洗。
3. 空内容或清洗后过短的 chunk 不入库。

建议流程：

```text
reader.load_data()
-> clean document text
-> splitter.get_nodes_from_documents()
-> clean chunk text
-> filter empty/noisy chunk
-> embedding
-> write blocks/chunks
```

### 6.3 PDF 解析器优化

第一阶段可以继续使用 `SimpleDirectoryReader`，但对 PDF 增加可替换解析层。

建议新增：

- `backend/app/rag/document_readers.py`

职责：

- 根据 `file_type` 或文件后缀选择解析器。
- PDF 优先使用 `PyMuPDF` 提取文本。
- 其他类型暂时沿用 `SimpleDirectoryReader`。

建议接口：

```python
def load_documents(local_path: str, file_type: str) -> list[LlamaDocument]:
    ...
```

PDF 解析策略：

- 使用 PyMuPDF `page.get_text("text")` 作为基础方案。
- 如遇表格密集文档，后续评估 `pdfplumber`。
- 如页面文本量极少但图片多，标记为疑似扫描件。

### 6.4 质量检测与日志

新增清洗质量统计：

- 原始字符数。
- 清洗后字符数。
- 删除控制字符数量。
- 修复异常空格次数。
- 空 chunk 过滤数量。
- 疑似乱码比例。

第一阶段可仅记录日志，不新增数据库字段。

后续可在 `documents` 表新增：

- `parse_quality_score`
- `parse_warning`
- `cleaning_stats`

### 6.5 已入库文档重建

该治理只能自动影响新入库或重新入库文档。

已有脏数据需要：

- 删除后重新上传。
- 或触发“重新入库”。
- 或提供批量 reingest 命令，对已有 `documents` 重新解析、清洗、切块、Embedding。

建议后续新增管理命令：

```powershell
python -m app.scripts.reingest_documents --kb-id <kb_id>
```

## 7. 技术路线

### 阶段一：低风险清洗层

目标：快速消除明显乱码和控制字符。

改造点：

- 新增 `text_cleaning.py`。
- 在 `ingest.py` 写入 chunk 前调用清洗函数。
- 对空 chunk 做过滤。
- 增加单元测试。

优点：

- 改动小。
- 不改变文件解析器。
- 对所有文件类型生效。

风险：

- 只能修复部分解析器输出问题。
- 对 PDF 阅读顺序、表格错乱帮助有限。

### 阶段二：文件类型解析器分层

目标：提升 PDF 等复杂文档的文本抽取质量。

改造点：

- 新增 `document_readers.py`。
- PDF 使用 PyMuPDF 专用读取逻辑。
- Office、TXT、Markdown 继续沿用当前能力。
- 增加解析失败 fallback。

优点：

- PDF 抽取质量更可控。
- 后续可扩展 OCR、表格解析、模板解析。

风险：

- 不同 PDF 类型效果差异较大。
- 需要更多样本文档验证。

### 阶段三：质量评估与重新入库工具

目标：可观测、可修复已有脏数据。

改造点：

- 增加解析质量日志。
- 增加文档重新入库命令或后台任务。
- 前端展示解析质量提示。

优点：

- 方便定位问题文档。
- 能治理历史数据。

风险：

- 需要更多工程投入。
- 重新入库会重新消耗 Embedding 成本。

## 8. 研发分解

### 8.1 后端改造

#### 新增文本清洗模块

文件：

- `backend/app/rag/text_cleaning.py`

内容：

- `normalize_extracted_text()`
- `remove_control_chars()`
- `repair_token_spacing()`
- `collapse_whitespace()`
- 清洗统计结构 `TextCleaningStats`

#### 调整入库逻辑

文件：

- `backend/app/rag/ingest.py`

改造：

- 解析后清洗文档文本。
- 切块后清洗 chunk 文本。
- 清洗后为空的 chunk 跳过。
- Embedding 只对清洗后文本生成。

#### 新增解析器选择层

文件：

- `backend/app/rag/document_readers.py`

改造：

- 封装当前 `SimpleDirectoryReader`。
- 为 PDF 预留 PyMuPDF 专用解析。
- 解析失败 fallback 到 `SimpleDirectoryReader`。

#### 日志与错误处理

- 对清洗前后字符数变化记录日志。
- 对疑似扫描件、低文本量 PDF 记录 warning。
- 对清洗后无有效文本的文档，标记为 `failed` 并写入错误信息。

### 8.2 前端改造

第一阶段无需改动前端。

中后期可增加：

- 文档列表展示解析质量状态。
- 切片详情展示“已清洗”标识。
- 文档详情展示解析警告。
- 重新入库按钮复用现有“切块配置保存并重新入库”逻辑或新增单独按钮。

### 8.3 测试改造

新增测试：

- 控制字符清理：

```text
abcdef -> abcdef
```

- 英文异常空格修复：

```text
adm in -> admin
```

- 账号异常空格修复：

```text
zlf.adm in-2107 -> zlf.admin-2107
```

- 邮箱异常空格修复：

```text
user @ example.com -> user@example.com
```

- 正常英文句子不被错误合并：

```text
admin can login -> admin can login
```

- 空白 chunk 过滤。
- 入库后 `chunks.content` 不包含控制字符。

## 9. 验收标准

### 9.1 功能验收

- 新上传文档入库后，切片内容不再出现 `\x01` 等控制字符。
- 示例文本 `zlf.adm in-2107` 可被清洗为：

```text
zlf.admin-2107
```

- 常见账号、邮箱、编号类文本不因异常空格导致检索失败。
- 清洗不破坏正常中文、英文句子、数字和标点。
- 清洗后为空的 chunk 不写入数据库。

### 9.2 检索验收

- 查询 `admin` 能命中包含原异常文本的文档。
- 查询 `zlf.admin-2107` 能命中清洗后的账号。
- RAG 引用片段不展示明显控制字符。

### 9.3 回归验收

- 文档上传、入库、切块、Embedding 流程仍正常。
- 修改切块配置并重新入库仍正常。
- 删除文档和删除知识库不受影响。
- 已有 RAG 问答接口不需要改调用方式。

## 11. 风险与注意事项

- 不应简单删除所有空格，否则会破坏正常英文句子。
- 不应只依赖前端展示替换，因为脏文本仍会影响检索和 Embedding。
- 清洗规则需要单元测试保护，避免误伤制度条款、编号、金额、邮箱和 URL。
- PDF 抽取没有单一完美方案，需要用真实业务样本文档验证。
- 已入库历史数据不会自动变干净，必须重新入库。
- 如果文档是扫描件，文本清洗无法替代 OCR。

---

## 12. 推荐实施策略

建议按低风险到高收益逐步实施：

1. 先做 `text_cleaning.py`，解决控制字符和高置信异常空格。
2. 接入 `ingest.py`，确保新数据干净。
3. 用问题样本文档验证 `zlf.adm in-2107` 是否清洗正确。
4. 再做 PDF 专用解析器，提高源头抽取质量。
5. 最后做历史文档重新入库和质量可观测。

第一阶段即可显著改善当前问题；要“彻底解决”，需要完成解析器分层、清洗规则、质量检测和历史数据重建四部分。
