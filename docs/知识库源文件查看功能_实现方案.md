# 知识库源文件查看功能实现方案

> 目标：在知识库文档查看界面中，同时查看源文件与切块结果，支持用户对照原文验证切片质量、解析效果和引用准确性。

---

## 1. 背景与现状

当前知识库页面已支持：

- 文档上传。
- 文档入库状态查看。
- 切块结果查看。
- 切块配置调整并重新入库。
- 文档重新入库。
- 文档删除。

当前问题：

- “查看”只能打开切片详情。
- 用户无法在同一界面查看原始文件。
- 当切片内容出现乱码、断词、缺页、表格错乱时，无法快速与源文件对照判断问题来源。
- 后端已有 MinIO 文件读取能力，但暂无源文件预览/下载 API。

---

## 2. 功能目标

### 2.1 核心目标

- 在文档操作区提供“查看”入口。
- 打开后可同时查看：
  - 源文件。
  - 切块结果。
- 支持在一个抽屉或弹窗中通过 Tab 切换“源文件 / 切块结果”。
- 源文件支持在线预览；不支持预览的文件提供下载。
- 所有源文件访问必须经过后端权限校验，不能直接暴露 MinIO 地址。

### 2.2 用户体验目标

- 用户点击文档“查看”后，默认进入“源文件”或“双栏查看”视图。
- 切片结果保留现有表格展示方式。
- PDF、图片、TXT/Markdown 优先在线预览。
- Word、Excel 等浏览器不易直接预览的文件，提供下载按钮。
- 用户可以快速在源文件与切片之间切换。

---

## 3. 范围与非范围

### 3.1 本次范围

- 新增源文件访问接口。
- 新增前端源文件预览能力。
- 改造现有切片 Drawer 为“文档查看 Drawer”。
- 在 Drawer 内增加 Tabs：
  - 源文件
  - 切块结果
- 对不支持预览的文件提供下载。

### 3.2 非范围

- 不做 Office 在线转 PDF。
- 不做源文件内容高亮定位到具体 chunk。
- 不做页码与 chunk 的双向联动。
- 不引入复杂文档标注系统。

这些能力可作为后续增强。

## 4. 后端设计

### 4.1 API 设计

#### 获取源文件

- `GET /documents/{doc_id}/source`
- 用途：返回源文件二进制流，供前端 iframe、object、图片标签或下载使用。
- 权限：用户必须对该文档所属知识库有访问权限。

Response Header：

```http
Content-Type: <document.file_type>
Content-Disposition: inline; filename*=UTF-8''<encoded_file_name>
Cache-Control: private, max-age=300
```

说明：

- 使用 `inline`，让浏览器优先预览。
- 对需要下载的文件，前端可通过 `<a download>` 或新接口控制。
- 文件名需要做 RFC 5987 编码，避免中文文件名乱码。

#### 下载源文件，可选

- `GET /documents/{doc_id}/download`
- 用途：强制下载源文件。

Response Header：

```http
Content-Disposition: attachment; filename*=UTF-8''<encoded_file_name>
```

第一阶段可以只实现 `/source`，前端下载时直接请求同一地址。

### 4.2 后端实现点

文件：

- `backend/app/api/routes_documents.py`

新增逻辑：

1. 根据 `doc_id` 查询文档。
2. 调用 `get_knowledge_base(session, doc.kb_id, current_user)` 做权限校验。
3. 解析 `doc.file_uri`：

```python
_, object_name = parse_minio_uri(doc.file_uri)
```

4. 从 MinIO 读取对象。
5. 使用 `StreamingResponse` 或 `Response` 返回文件内容。

当前 `MinioStorage` 已有：

```python
download_to_bytes(object_name: str) -> bytes
```

第一阶段可直接使用 `download_to_bytes()`，实现简单。

如果未来支持大文件，建议新增流式方法：

```python
def stream_object(self, object_name: str) -> Iterator[bytes]:
    ...
```

### 4.3 文件大小策略

当前上传已有大小限制：

- `settings.ingest_max_file_mb`

第一阶段可直接读取字节返回。

后续优化：

- 大于指定阈值时使用流式响应。
- 支持 HTTP Range，提升 PDF 预览体验。

### 4.4 安全策略

- 不返回 MinIO 真实地址。
- 不使用公开 bucket。
- 每次源文件访问都经过登录态和知识库权限校验。
- 禁止通过 `file_uri` 参数访问任意对象，只允许通过 `doc_id` 间接访问。
- 返回 `Content-Type` 时使用数据库记录的 `file_type`，缺失时 fallback 到 `application/octet-stream`。

## 5. 前端设计

### 5.1 API 封装

文件：

- `frontend/src/api/documents.ts`

新增：

```ts
export function getDocumentSourceUrl(docId: string): string {
  return `/documents/${docId}/source`;
}
```

如果 axios baseURL 可能非空，建议封装：

```ts
export function getDocumentSourceUrl(docId: string): string {
  return `${apiBaseUrl}/documents/${docId}/source`;
}
```

注意：

- 如果源文件通过 iframe/object 加载，浏览器请求需要带鉴权。
- 当前系统使用 `Authorization: Bearer` 请求头，iframe 无法自动带 axios interceptor。

因此推荐两种方案：

### 5.2 鉴权方案选择

#### 方案 A：前端 fetch Blob，再创建 object URL

流程：

1. 使用 axios 请求 `/documents/{doc_id}/source`。
2. `responseType: "blob"`。
3. 使用 `URL.createObjectURL(blob)` 生成临时预览地址。
4. 用 iframe/img/object 显示 blob URL。
5. Drawer 关闭时 `URL.revokeObjectURL()`。

优点：

- 复用现有 Authorization header。
- 不需要改 token 存储方式。
- 不暴露 MinIO。

缺点：

- 文件会先完整下载到浏览器内存。
- 大文件预览体验不如 Range 流式。

第一阶段推荐方案 A。

#### 方案 B：后端生成短期预签名 URL

后端返回短期有效 URL，前端 iframe 直接加载。

优点：

- 浏览器原生预览体验更好。
- MinIO 可直接处理大文件。

缺点：

- 需要生成和管理短期 URL。
- 需要仔细控制有效期和访问范围。

本次不建议优先采用。

### 5.3 前端页面改造

文件：

- `frontend/src/pages/KnowledgeBase/KnowledgeBasePage.tsx`

当前：

- `openChunks(doc)` 打开 Drawer。
- Drawer title 为 `切片详情 - 文件名`。
- Drawer 内只显示切片 Table。

建议改造：

- 将 `chunkDrawerOpen` 语义调整为 `documentViewerOpen`。
- `openChunks(doc)` 改为 `openDocumentViewer(doc)`。
- Drawer 标题改为：

```text
文档查看 - <file_name>
```

- Drawer 内加入 Tabs：
  - `源文件`
  - `切块结果`

### 5.4 源文件预览组件

建议新增组件：

- `frontend/src/components/DocumentPreview/DocumentPreview.tsx`

Props：

```ts
interface DocumentPreviewProps {
  document: DocumentItem | null;
}
```

职责：

- 拉取源文件 blob。
- 根据 `file_type` 判断预览方式。
- 展示加载态、错误态和下载按钮。
- 关闭或切换文件时释放 object URL。

### 5.5 文件类型预览策略

| 文件类型 | 预览方式 | 说明 |
|---|---|---|
| `application/pdf` | `iframe` | 浏览器原生 PDF 预览 |
| `image/*` | `img` | 图片预览 |
| `text/plain` | `pre` 或 iframe | 可直接展示文本 |
| `text/markdown` | `pre` | 第一阶段不渲染 Markdown |
| `application/json` | `pre` | 格式化展示 |
| Word | 下载 | 浏览器不稳定支持 |
| Excel | 下载 | 不做在线表格预览 |
| 其他 | 下载 | 显示“不支持在线预览” |

### 5.6 UI 布局建议

第一阶段采用 Tabs：

```text
Drawer: 文档查看 - xxx.pdf

[源文件] [切块结果]

源文件 Tab:
  - 顶部：文件名、类型、大小、下载按钮
  - 内容：PDF iframe / image / text preview / unsupported fallback

切块结果 Tab:
  - 复用当前 Table
```

后续增强可做左右双栏：

```text
左侧源文件 | 右侧切块结果
```

但第一阶段 Tabs 更稳，改动小，移动端也更容易适配。

## 6. 状态管理

新增状态建议：

```ts
const [viewerOpen, setViewerOpen] = useState(false);
const [viewerTab, setViewerTab] = useState<"source" | "chunks">("source");
const [sourceLoading, setSourceLoading] = useState(false);
const [sourceObjectUrl, setSourceObjectUrl] = useState<string | null>(null);
const [sourceError, setSourceError] = useState<string | null>(null);
```

打开文档：

```ts
openDocumentViewer(doc) {
  setActiveDocument(doc);
  setViewerTab("source");
  setViewerOpen(true);
  loadChunks(doc.id);
  loadSourceBlob(doc.id);
}
```

关闭 Drawer：

```ts
URL.revokeObjectURL(sourceObjectUrl)
setSourceObjectUrl(null)
setChunkItems([])
setActiveDocument(null)
```

## 7. 测试计划

### 7.1 后端测试

- 有权限用户可访问 `/documents/{doc_id}/source`。
- 无权限用户访问返回 `403`。
- 不存在文档返回 `404`。
- 返回 `Content-Type` 与 `document.file_type` 一致。
- 返回内容与 MinIO 原文件一致。

### 7.2 前端测试

- 点击“查看”打开文档查看 Drawer。
- 默认展示源文件 Tab。
- PDF 文件使用 iframe 预览。
- 图片文件使用 img 预览。
- 不支持在线预览的文件显示下载入口。
- 切换到“切块结果”后展示现有切片表格。
- 关闭 Drawer 后释放 blob URL。

### 7.3 回归测试

- 上传文档仍正常。
- 查看切块仍正常。
- 重新入库仍正常。
- 删除文档仍正常。
- 权限不足用户不能绕过接口查看源文件。

## 9. 验收标准

- 用户点击文档“查看”后，可以看到源文件。
- 用户可以在同一 Drawer 中切换查看切块结果。
- PDF、图片、文本文件可在线预览。
- 不支持在线预览的文件可下载。
- 源文件访问经过后端权限校验。
- 不暴露 MinIO 内部地址。
- 现有切片查看、重新入库、删除功能不受影响。

## 10. 推荐实施策略

建议按以下顺序实现：

1. 后端先提供 `/documents/{doc_id}/source`。
2. 前端通过 axios 拉取 blob，生成 object URL。
3. 新增 `DocumentPreview` 组件。
4. 将现有“查看切片”改为“文档查看”，用 Tabs 同时承载源文件和切块结果。
5. 验证 PDF、图片、TXT、Word/Excel 下载场景。

第一阶段优先做“可安全查看源文件 + 可切换切块结果”，暂不做复杂的源文件与 chunk 高亮联动。
